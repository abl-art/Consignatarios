Arquitectura del sistema

Sección 1 de 4 — Visión general de componentes y flujos principales

Frontend (Next.js 14 + Tailwind)
Panel Admin (GOcelular)
Dashboard global · Asignación de IMEI · Sync gocuotas · Reportes
Panel Consignatario
Su propio stock · Control de inventario periódico · Solo lectura de ventas
Backend (Next.js API Routes + Supabase)
Supabase (DB + Auth)
PostgreSQL · RLS por consignatario · Auth con roles
Cron Job T-1 (Vercel)
Consulta gocuotas DB · Marca IMEIs como vendidos · Actualiza saldos
Sistema externo — Gocuotas (T-1)
Base de datos de respaldo consultada diariamente. Endpoint: query por owner_id o store_id → devuelve ventas del día anterior con IMEI de cada equipo vendido.
Flujos principales
📦 Asignación de stock
Admin carga IMEI(s) → selecciona consignatario → confirma entrega → cada IMEI queda en estado asignado
🔄 Sync gocuotas (diario)
Cron job a las 6am → consulta ventas T-1 → matchea IMEI → marca como vendido → actualiza stock disponible
🔍 Control de inventario
Consignatario inicia auditoría → escanea/confirma IMEIs presentes → sistema detecta faltantes → genera diferencias
⚠️ Diferencias de inventario
IMEI asignado + no vendido en gocuotas + no presente en auditoría = deuda del consignatario por ese IMEI
