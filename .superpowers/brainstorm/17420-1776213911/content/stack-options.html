¿Qué stack usamos?

Las 3 opciones para una app web con soporte mobile — con trade-offs reales

A

⭐ Next.js + Supabase + PWA RECOMENDADO

App web de alta calidad deployada como PWA (Progressive Web App) — instalable en celular desde el navegador sin pasar por una app store.

Pros

  • Mejor UX de escritorio de las 3 opciones
  • Server-side rendering → ideal para el sync T-1 de gocuotas
  • PWA: instalable en Android/iOS desde Chrome/Safari
  • Supabase ya lo conocés
  • Deploy en Vercel gratis

Contras

  • No está en App Store / Play Store (es PWA, no nativa)
  • Stack nuevo (no es Expo)
B

Expo (React Native Web) + Supabase

El mismo stack de tu app proveedores-app actual. Un solo código corre en web, iOS y Android.

Pros

  • Ya lo conocés y lo tenés configurado
  • Nativo real en iOS y Android
  • Un solo codebase

Contras

  • Web UX inferior — React Native Web no está pensado para dashboards complejos
  • La imagen de referencia es un panel de escritorio que no queda bien en RN Web
  • Requiere build nativo para subir a stores
C

React + Vite + Capacitor + Supabase

SPA web que se compila a app nativa mediante Capacitor (como Ionic). Lo mejor de los dos mundos en teoría.

Pros

  • Web UX excelente
  • Compilación nativa real para stores

Contras

  • Más complejo de mantener (web + build nativo)
  • Sin SSR → sync T-1 más complicado
  • Menos adoptado en el ecosistema actual

Mi recomendación: Opción A — Next.js + PWA. El panel de escritorio de la imagen de referencia se ve y funciona perfectamente en Next.js. En mobile, la PWA es instalable y funciona offline. El sync T-1 con gocuotas se maneja con un API Route o cron job de Vercel. Supabase ya lo tenés configurado.
